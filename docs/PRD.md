# banars モバイルアプリ PRD

## 背景と目的
- **課題認識**: banars は草野球チームとして活動しているが、出欠管理が口頭やチャットベースに依存しており、運営側の確認コストが高い。特に「誰が出席予定か瞬時にわからない」「期限内に未回答者が発生しリマインドが必要」「イベント詳細を参加者にまとめて提示できない」といった運用上の摩擦がある。
- **解決方針**: モバイルアプリ化により、メンバーが自発的に出欠を入力できる UX を用意し、出欠状況の可視化とリマインドの負担を軽減する。運営の負荷を下げることが主目的だが、同時にメンバーの利便性向上を図ることで入力率を高める狙いがある。

## ステークホルダーとユーザー
- **共通アプリ利用**: 運営側もメンバー側も同じアプリを利用し、情報格差をなくす。運営専用アプリを作らないのは、管理対象が少人数でありツールを一本化する方が運用コストを抑えられるため。
- **運営ロール**: 運営担当のみが使える機能（試合やイベントの登録、メンバー管理など）を設ける。経理やスケジュール管理など運営特有の作業があり、誤操作防止のためにも権限管理が必要という背景による。
- **主要ユースケース**:
  - 運営が試合/イベントを作成する。
  - すべてのメンバーが登録済みイベントに対して出欠（出席/欠席/保留）を入力し、状況を閲覧する。

## 要件（機能・体験）
### 試合・イベント登録（Must）
- **日時と場所を必須項目にする背景**: 試合実施可否や準備を判断する上で、最も重要かつ誰もが確認したい情報のため。入力漏れを防ぎ、イベント一覧でも同情報を軸にソート/表示できるようにする。
- **地図埋め込み**: 場所を登録した際に Google Maps 等の地図をアプリ内でプレビュー表示する。遠征先など初めて行く場所でも即座にアクセス方法を把握しやすくする狙い。
- **自由入力欄**: 対戦相手、集合時間、参加費などは都度変わるため、必須項目にすると運営の入力手間が増える。自由記述にして柔軟性と記載速度を担保する。

### 出欠入力（Must）
- **ステータス**: 出席 / 欠席 / 保留の 3 種。草野球チームでは「調整中」のケースが一定数あり、二択では運営判断が難しいため保留を設ける。
- **理由入力**: 欠席や保留の理由を記録できるようにすることで、運営によるリマインドの要否や代替調整の判断に使える。自由記述欄として実装予定。

### メンバー登録と認証
- **Signup（Must）**: 新メンバーをアプリに onboard させるために必須。最初に運営が招待リンクを発行し、メンバー自身がプロファイルを作成できるようにする想定。
- **LINE ログイン（Should）**: チーム内で LINE 利用が定着しており、既存の認証情報を使いたいニーズが強い。メール/パスワード方式よりも入力負担を減らし、ログイン率を上げる狙いで LINE 連携を採用する。

### 優先度まとめ
- **Must**: 試合/イベント登録（日時・場所必須、地図埋め込み、自由入力）、出欠登録（出席/欠席/保留＋理由入力）、メンバー signup。
- **Should**: LINE ログイン連携。
- **Could**: 今回は特に設定しない。運営コスト削減が最優先なため Must 項目の実現を重視する。
