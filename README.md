# banars-app

banars 草野球チームの運営を効率化するモバイルアプリのリポジトリです。

## ドキュメントの役割
- README.md は人間向けの案内です。運用ルールや共有事項はここに明文化します。
- AGENTS.md は AI エージェント向けのルール集です。人が読む想定の説明は README.md に寄せてください。
- PRD は `docs/PRD.md`、設計ドキュメント群は `docs/DESIGN_DOCS/` 配下で管理します。

## PRD 修正時の心得
- **章立ての維持**: 小規模アプリのため PRD（`docs/PRD.md`）は「背景と目的」「ステークホルダーとユーザー」「要件（機能・体験）」のみで構成します。4〜7 章は追加しません。
- **決定理由の明記**: 重要な選定には必ず「なぜその選択をしたのか」の背景を添えます。運営課題やユーザー体験と結びつけることで後追いしやすくします。
- **DESIGN_DOCS との分担**: PRD は「何を達成したいか」「どの価値を提供したいか」まで。技術的な実装方法は DESIGN_DOCS に記載し、責務を混在させないようにします。
- **対話的な更新**: 要件が不明確な場合は、関係者にヒアリングするか、同様の Q&A を重ねて情報を確定させてから PRD を編集します。PRD の改訂は常に人と対話しながら進め、独断で改変しないでください。

この運用を守ることで、PRD を読むだけで意思決定の背景が理解でき、設計ドキュメントとの整合性も保ちやすくなります。

## DESIGN_DOCS の管理方針
- **フォルダ構成**: `docs/DESIGN_DOCS/overview.md` に全体アーキテクチャとコンテキスト一覧を集約。各コンテキストは `docs/DESIGN_DOCS/{context}/` ディレクトリを作り、その中に機能単位の Markdown（必要に応じて `index.md`）を配置します。
- **命名と粒度**: コンテキストディレクトリ名は英小文字スネークケースで統一し、同一コンテキストの機能群をひとまとめにします。ファイル名には機能を表す英単語を使い、一覧を見ただけで内容が想起できるようにします。
- **記載内容**: 各ファイルでは、実装方針・依存サービス・権限設計・データモデルを記述し、決定理由も必ず添えます。
- **更新手順**: コンテキストや機能を追加/変更した際は、当該ファイルに加え `overview.md` のリンク・説明も同時に更新し、ドキュメントナビゲーションを最新に保ちます。DESIGN_DOCS の修正も、必ず人と対話しながら内容を確認した上で実施してください。
